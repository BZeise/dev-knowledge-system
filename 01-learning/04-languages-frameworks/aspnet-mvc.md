# ASP.NET MVC Framework

## MVC Architecture Overview

**MVC (Model-View-Controller)** separates an application into three components:

- **Model** — Data and business logic
- **View** — UI/presentation (HTML, CSS)
- **Controller** — Receives input, coordinates between Model and View, returns a response

This separation enables:
- Independent development of each layer
- Easier testing
- Clear responsibilities
- Code reusability

---

## ViewData vs ViewBag vs TempData

All three pass data from a controller action to a view in ASP.NET MVC.

| Aspect | ViewData | ViewBag | TempData |
|--------|----------|---------|----------|
| **Type** | Dictionary<string, object> | dynamic | Dictionary<string, object> |
| **Lifetime** | Current request only | Current request only | Survives one redirect |
| **Casting** | Requires casting when reading | No casting needed | Requires casting |
| **Use Case** | Passing data to view | Passing data to view | Post-redirect-get pattern |

### ViewData Example

```csharp
// In controller
ViewData["Message"] = "Hello, World!";
ViewData["User"] = new User { Name = "John" };

// In view
<p>@ViewData["Message"]</p>
<p>@((User)ViewData["User"]).Name</p>  <!-- Requires casting -->
```

### ViewBag Example

```csharp
// In controller
ViewBag.Message = "Hello, World!";
ViewBag.User = new User { Name = "John" };

// In view
<p>@ViewBag.Message</p>
<p>@ViewBag.User.Name</p>  <!-- No casting needed -->
```

**ViewBag is syntactic sugar over ViewData** — they share the same underlying dictionary.

### TempData Example

```csharp
// In controller action 1
public ActionResult Create(User user)
{
    db.Users.Add(user);
    db.SaveChanges();
    
    TempData["SuccessMessage"] = "User created successfully!";
    return RedirectToAction("Details", new { id = user.Id });
}

// In controller action 2
public ActionResult Details(int id)
{
    var user = db.Users.Find(id);
    return View(user);
}

// In view
@if (TempData["SuccessMessage"] != null)
{
    <div class="alert alert-success">
        @TempData["SuccessMessage"]
    </div>
}
```

TempData persists through **one redirect**, useful for the post-redirect-get pattern. After being read, it's marked for deletion.

## Validation in MVC

### Data Annotations

Attributes on model properties define validation rules:

```csharp
public class UserModel
{
    [Required]
    [StringLength(100, MinimumLength = 2)]
    public string Name { get; set; }

    [EmailAddress]
    public string Email { get; set; }

    [Range(18, 120)]
    public int Age { get; set; }
    
    [RegularExpression(@"^\d{3}-\d{3}-\d{4}$")]
    public string PhoneNumber { get; set; }
}
```

### ModelState.IsValid

Check in the controller action before acting on submitted data:

```csharp
[HttpPost]
public ActionResult Create(UserModel model)
{
    if (ModelState.IsValid)
    {
        // Process valid data
        db.Users.Add(model);
        db.SaveChanges();
        return RedirectToAction("Index");
    }
    
    // Return view with validation errors
    return View(model);
}
```

If `ModelState.IsValid` is false, return the view with validation errors displayed.

### Client-Side Validation

Automatically generated by including jQuery Validation scripts.

```html
<script src="~/Scripts/jquery.validate.min.js"></script>
<script src="~/Scripts/jquery.validate.unobtrusive.min.js"></script>
```

Validates in the browser before submit.

### Server-Side Validation

**Always runs regardless of client-side validation.**

Never trust the client alone. Always validate on the server.

## Web API vs RESTful Services

### Web API
The general term for any API communicating over HTTP/HTTPS.

### REST (Representational State Transfer)
A specific architectural style with defining constraints:
- **Stateless** — Each request contains all context needed
- **Client-server** — Clear separation of concerns
- **Uniform interface** — Resources identified by URIs; standard HTTP verbs communicate intent
- **Resource-based** — Operating on nouns, not calling action-verbs

### REST API Example

```csharp
// RESTful API
[ApiController]
[Route("api/[controller]")]
public class UsersController : ControllerBase
{
    // GET /api/users — list all
    [HttpGet]
    public ActionResult<IEnumerable<User>> GetAll() { }
    
    // GET /api/users/5 — get specific
    [HttpGet("{id}")]
    public ActionResult<User> GetById(int id) { }
    
    // POST /api/users — create
    [HttpPost]
    public ActionResult<User> Create([FromBody] CreateUserRequest request) { }
    
    // PUT /api/users/5 — update
    [HttpPut("{id}")]
    public ActionResult<User> Update(int id, [FromBody] UpdateUserRequest request) { }
    
    // DELETE /api/users/5 — delete
    [HttpDelete("{id}")]
    public ActionResult Delete(int id) { }
}
```

### Non-RESTful Web API (Anti-pattern)

```csharp
// NOT RESTful — action-based URLs and verbs in names
[HttpPost("/api/getUser")]
public User GetUser(int id) { }

[HttpPost("/api/updateUser")]
public void UpdateUser(User user) { }

[HttpPost("/api/deleteUser")]
public void DeleteUser(int id) { }
```

### Key Difference

**All REST APIs are Web APIs; not all Web APIs are REST.**

An API that uses HTTP but doesn't follow REST constraints (e.g., uses POST for everything, has action-based URLs) is a Web API but **not RESTful**.

## HTTP Status Codes

| Code | Meaning | Use When |
|------|---------|----------|
| **2xx Success** | Request succeeded | |
| 200 | OK | Request succeeded, returning data |
| 201 | Created | Resource created successfully |
| 204 | No Content | Request succeeded, no content to return |
| **4xx Client Error** | Client's fault | |
| 400 | Bad Request | Invalid request format or parameters |
| 401 | Unauthorized | Authentication required |
| 403 | Forbidden | Authenticated but not authorized |
| 404 | Not Found | Resource doesn't exist |
| 409 | Conflict | Request conflicts with current state |
| **5xx Server Error** | Server's fault | |
| 500 | Internal Server Error | Unexpected server error |
| 503 | Service Unavailable | Server temporarily unavailable |

## CORS (Cross-Origin Resource Sharing)

A browser security mechanism that restricts web pages from making requests to a different origin (scheme + domain + port) than the one that served the page.

### The Problem

```
Browser error:
Access to fetch at 'https://api.example.com' 
from origin 'https://myapp.com' 
has been blocked by CORS policy
```

### The Solution

The **server must include Access-Control headers**:

```csharp
// ASP.NET Core
builder.Services.AddCors(options => {
    options.AddPolicy("AllowMyFrontend", policy =>
        policy.WithOrigins("https://myapp.com")
              .AllowAnyMethod()
              .AllowAnyHeader()
              .AllowCredentials());
});

app.UseCors("AllowMyFrontend");
```

### Important Note

**CORS is a browser enforcement** — non-browser clients (Postman, curl, server-to-server) are **not subject to it**.

## Dependency Injection in ASP.NET Core

ASP.NET Core has built-in dependency injection:

```csharp
// Register services in Program.cs
builder.Services.AddScoped<IUserRepository, UserRepository>();
builder.Services.AddScoped<IUserService, UserService>();

// Inject into controller
[ApiController]
[Route("api/[controller]")]
public class UsersController : ControllerBase
{
    private readonly IUserService _userService;
    
    public UsersController(IUserService userService)
    {
        _userService = userService;
    }
    
    [HttpGet("{id}")]
    public ActionResult<User> GetUser(int id)
    {
        return _userService.GetUser(id);
    }
}
```
